# macOS desktop builds

Polymons Player and Poly Studio are packaged for both Intel (`x64`) and Apple
Silicon (`arm64`) by `.github/workflows/package-desktop.yml`. macOS packages
must be produced on the GitHub macOS runner; the Windows development machine
can compile the shared TypeScript but cannot create or validate a trustworthy
DMG.

## Test packages

1. Open **Actions > Package Desktop Apps > Run workflow** in GitHub.
2. Download the `player-mac` and `studio-mac` workflow artifacts.
3. Install the DMG matching the Mac's architecture.
4. Open Polymons Player once so macOS registers the `polymons://` protocol.
5. Verify website launches, Studio playtests, keyboard input, audio, and updates.

The workflow validates both architectures, DMG metadata, ZIP integrity, app
property lists, and Player protocol registration before uploading artifacts.

## Signing and notarization

Unsigned workflow artifacts are suitable for internal testing. For public
distribution, add these encrypted GitHub Actions secrets:

- `MACOS_CSC_LINK`: Developer ID Application certificate exported as a `.p12`
  file or its base64 representation.
- `MACOS_CSC_KEY_PASSWORD`: password used to export the certificate.
- `APPLE_ID`: Apple developer account email.
- `APPLE_APP_SPECIFIC_PASSWORD`: app-specific password for notarization.
- `APPLE_TEAM_ID`: Apple developer team identifier.

When the signing secret is present, CI also performs strict signature
verification. Electron Builder uses the Apple credentials to notarize the
DMGs. Never commit certificates, passwords, or Apple credentials to the repo.

## Releases and updates

Tagged builds embed `github.ref_name` in both apps. The macOS updater compares
that tag with the latest GitHub release, preventing the installed patch from
being downloaded repeatedly. Release tags should therefore be unique and the
DMGs generated by the same tagged workflow must be attached to that release.
